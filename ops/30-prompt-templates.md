# 30-prompt-templates：派工 prompt 模板

> 讀者：主對話模型。用法：複製對應模板、填掉所有【】、刪掉不適用的可選行，整段作為 subagent 的 prompt。
> 不要即興寫派工 prompt。模板缺什麼欄位，先補模板（走 `40-maintenance.md`），再派工。
> 所有模板共用的回報鐵律：第一行必為 DONE / BLOCKED / FAILED（唯一例外：模板 5 驗收用 PASS / FAIL）；長產物落檔傳路徑；禁止貼原文超過 20 行。

---

## 模板 1：搜尋（掃 repo / 找定義 / 查資料位置）

```
[目標] 在【範圍：目錄或 repo】找出【要找的東西】。
[動機] 主任務是【一句話】，需要這個資訊來【用途】。
[做法提示] 建議從【關鍵字 / 入口檔案】下手。（可選）
[驗收條件]
- 每個結果附 檔案:行號。
- 若找不到，列出實際搜過的關鍵字與目錄，明確回報「不存在」而非沉默。
[回報格式]
第一行：DONE/BLOCKED/FAILED
結果清單：檔案:行號 — 一句話說明這裡是什麼
（若結果超過 10 條：落檔到【路徑】，回報只給路徑 + 前 3 條範例）
[禁止] 貼出任何超過 20 行的原文。
```

## 模板 2：實作（新功能 / 新檔案）

```
[目標] 實作【功能一句話】，產出位置【檔案路徑】。
[動機] 【為什麼要這個功能，使用情境一句話】。遇到規格沒寫到的細節，按這個動機取捨並在回報中列出你做的假設。
[規格]
- 輸入：【】 輸出：【】
- 必須遵守：【專案慣例/介面約定/CLAUDE.md 硬規則編號】
- 不要做：【明確排除的範圍，防止過度工程】
[驗收條件]
- 【測試指令】全綠，含至少【N】個新增測試（覆蓋 happy path + 【指定的 edge case】）。
- 不得修改【不准動的檔案/測試】。
[回報格式]
第一行：DONE/BLOCKED/FAILED
改動清單：檔案:行號範圍 — 改了什麼
測試證據：實跑輸出的最後 5 行
假設清單：規格沒寫、我自己決定的事項
```

## 模板 3：重構（改既有程式碼，行為不變）

```
[目標] 重構【範圍】：【要達成的結構變化一句話】。
[動機] 【現在的痛點】。
[鐵律]
- 行為不變：重構前先跑【測試指令】記錄基準，重構後必須同樣全綠。
- 改前備份或確認 git 乾淨（`git status`），可回退才准動手。
- 一次只做這個重構，發現其他該修的東西記下來回報，不要順手修。
[驗收條件]
- 測試前後皆綠（附兩次輸出）。
- 【結構驗收：例如「X 類不再依賴 Y 模組」，用 grep 可證明】
[回報格式]
第一行：DONE/BLOCKED/FAILED
基準測試輸出末 3 行 / 重構後測試輸出末 3 行
改動清單：檔案:行號範圍
順手發現但沒動的問題清單
```

## 模板 4：研究（查網頁 / 讀文件 / 比較方案）

```
[目標] 回答這個問題：【具體問題，可判斷有沒有答到】。
[動機] 這個答案會用來決定【下游決策】，所以【最在乎的維度：例如相容性>效能】。
[範圍] 查【來源範圍：官方文件優先/指定網站/repo 內文件】。截止條件：【最多 N 個來源或找到明確答案為止】。
[驗收條件]
- 每個關鍵結論附來源（URL 或 檔案:行號）。
- 查不到的部分明確標「未查證」，禁止腦補。
- 來源之間有衝突時，並列衝突而不是私自裁決。
[回報格式]
第一行：DONE/BLOCKED/FAILED
結論（3 句以內直接回答問題）
依據：來源清單，每條一行
未查證/衝突事項
（完整筆記落檔到【路徑】，回報不貼全文）
```

## 模板 5：審查 / 驗收（fresh-context，驗別人的產出）

```
[身分] 你是驗收者，與產出者無關。你的價值在找出問題，「全部通過」的回報必須經得起追問。
[目標] 驗收【產物路徑】是否滿足下列條件。
[原始驗收條件]（從派工 prompt 原文複製，逐條列出）
1. 【】
2. 【】
[驗法]
- 程式碼：實跑【測試指令】，不接受「看起來會過」。
- 檔案：實際打開 read-back，檢查殘留【填空】、斷掉的引用、自相矛盾。
- 另外以找碴立場列出至少 3 個質疑點並逐一查證（查證後排除也算，但要寫查了什麼）。
[回報格式]
第一行：PASS / FAIL
逐條：條件 N — pass/fail — 證據（檔案:行號 或 實跑輸出末行）
質疑點清單與查證結果
（FAIL 時：具體差在哪、建議的修法方向一句話）
```
