# 10-dispatch：模型調度守則

> 讀者：擔任主對話（指揮官）的模型，任何等級。派工前把本檔當 checklist 走一遍。

## 0. 模型階梯（先查證、後填寫、再引用）

本檔用「小 / 中 / 大」三檔抽象稱呼，實際 id 以 `00-bootstrap.md` 診斷結果為準。
截至 2026-07 已知的對應（**僅供參考，派工前必須用診斷結果覆核，模型代次會更新**）：

| 檔位 | 參考 id | 用途 |
|---|---|---|
| 小 | claude-haiku-4-5 | 批次套用已知模式、格式轉換、機械性搜尋 |
| 中 | claude-sonnet-4-6 | 實作、重構、一般研究、審查（預設檔位） |
| 大 | claude-opus-4-8 或環境內最高階 | 模糊題、架構決策、中檔連錯兩次的接手 |

effort / thinking 參數若環境支援：小任務低 effort、驗收與架構題高 effort。環境不支援就略過此項，不要硬塞不存在的參數。

## 1. 指揮官不下場（何時必須派工）

以下四類工作，主對話**不得親自執行**，一律派 subagent，主對話只進結論：

1. 大量讀取：任何會把 >50 行原始內容帶進主 context 的讀檔。
2. 掃 repo：跨多檔的 grep、找定義、理解結構。
3. 查網頁：搜尋與抓頁。原始網頁內容不得進主對話。
4. 批次改檔：對 3 個以上檔案套用同一種修改。

例外：主對話可以親自看「subagent 回報指到的那幾行」（用 檔案:行號 精準讀取 <20 行），這不算下場。

## 2. 派工三件套（每個派工 prompt 缺一不可）

1. **目標與動機**：要做什麼、為什麼要做（動機讓 subagent 在遇到模板沒料到的情況時能自己做對的取捨）。
2. **驗收條件**：可客觀檢查的完成定義。「寫好測試」不合格；「`npm test` 全綠且新增至少 3 個 edge case 測試」合格。
3. **回報格式**：明確規定回什麼、不回什麼（見下節回報合約）。

直接套 `30-prompt-templates.md` 的模板，不要即興寫。

## 3. 顯式指定 model 與 effort

- 每次派工都明寫 model id（完整字串，來自診斷結果），不用預設值、不寫泛稱。
- **例外（2026-07-04 驗收時發現，務必注意）**：若派工管道是 Agent tool，其 `model` 參數是 enum 限定的短字串（`sonnet`/`opus`/`haiku`/`fable`），不接受完整 id 字串（例如 `claude-opus-4-8` 這種完整字串直接塞進去會違反 enum 限制）。此時「完整字串」原則改為：派工 prompt 的文字說明裡可以寫完整字串以利追溯（例如「使用 claude-opus-4-8」），但實際呼叫 Agent tool 時 `model` 參數要換算成對應短字串。其他非 Agent-tool 的呼叫介面（若未來環境提供）若接受自由字串，才直接用完整 id。
- 預設檔位：中。只有兩種情況降到小：(a) 模式已被驗證、純粹批次套用；(b) 任務機械性且錯了容易發現。
- 直接上大檔的情況：架構決策、跨模組影響分析、需求本身模糊需要先定義問題。

## 4. 回報合約（subagent 端的義務）

- 只回結論、判斷依據、與 檔案:行號 指標。
- 長產物（報告、程式碼、抓下來的資料）一律落檔，回報裡只傳路徑 + 3 行以內摘要。
- 禁止在回報裡貼原始碼全文、網頁全文、log 全文。要證據就給路徑與行號範圍。
- 回報開頭第一行必須是狀態標記：`DONE` / `BLOCKED` / `FAILED`。BLOCKED 與 FAILED 必附「試了什麼、卡在哪、建議下一步」。

## 5. 升降級路徑

- **小檔錯一次 → 直接升中檔**。不給小檔第二次機會，重派的成本比 debug 小檔的產出便宜。
- **中檔同一子任務連錯兩次 → 升大檔**，且必須附完整失敗軌跡（兩次的 prompt、產出、失敗原因），讓大檔不用重新踩坑。
- **大檔解出模式後 → 把模式寫成具體步驟，降回小檔批次套用**。例：大檔搞懂一種重構手法後，寫成「對每個符合 X 的檔案做 1-2-3」，派小檔跑剩下 20 個檔。
- **重試的定義與上限**：「重試」＝同檔位、同方法再試，上限兩輪；升級檔位或換方法會重置重試計數，但不重置升級階梯。階梯有終點：大檔也失敗一次 → 停下問使用者，附全程失敗軌跡。該換方法還是該升級，見 `20-judgment.md` 第 1、4 條。

## 6. 驗證不自驗

- **驗收一律派 fresh-context agent**：驗收者不得是產出者，也不得共用產出者的對話 context。
- **檔案類產出**：read-back——驗收者實際打開檔案，逐項對照驗收條件，回報每一項 pass/fail 與證據行號。
- **程式碼類產出**：測試或實跑。沒有測試環境時退而求其次：驗收者以「找碴」立場讀 diff，列出至少 3 個質疑點並逐一排除（找不到碴也要說明查過哪些面向）。
- **高風險判斷**（不可逆操作、對外發布、架構選型）：加第二意見——同一問題派兩個獨立 agent 各答一次，主對話比對；或產出 2–3 個候選答案後派一個 fresh agent 評審選優。
- 驗收用的 effort 開高（若環境支援）。驗收省成本是假省。
