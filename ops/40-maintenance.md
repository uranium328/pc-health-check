# 40-maintenance：制度維護協議

> 讀者：未來任何要修改本制度檔案的模型。改之前把本檔讀完。

## 0. 通用鐵律

- 改任何檔案前先備份：`cp <檔> <檔>.bak-$(date +%Y%m%d)`。
- 改完必做 read-back：重開檔案確認改動落地、沒弄壞其他段落、引用路徑仍有效。
- 每次修改在被改檔案底部的變更記錄區（或 lessons.md）留一行：日期、改了什麼、為什麼。

## 1. 權限分級

### 可自行修改（不需問使用者）
- `lessons.md`：追加教訓（append-only，見第 2 節格式）。
- CLAUDE.md 的「環境事實」區與路由表**新增**一行。
- `30-prompt-templates.md`：在既有模板上**新增**欄位或新增模板。
- `00-bootstrap.md` 的診斷結果區更新。
- 修正明確的事實錯誤：錯的路徑、錯的指令、已改名的工具（修正時在變更記錄註明證據）。

### 動之前必須先問使用者
- **刪除或放寬**任何硬規則、驗收要求、重試上限、升降級門檻。（收緊可以自行做，放寬不行——放寬通常是模型想省事的訊號。）
- 刪除路由表既有條目、刪除任何 ops/ 檔案。
- 改 `20-judgment.md` 的任何判準本體（新增正反例可以自行做）。
- 刪除 `20-judgment.md` 第 6 節（誠實條款）——任何情況下都不准刪，連問都不用問，答案是不行。

## 2. 踩坑後教訓寫回哪裡、什麼格式

坑一踩到，當下就寫進 `lessons.md`（不要等任務結束，session 隨時會斷）。格式固定四行：

```
## L-【流水號】 【日期】 【標籤：dispatch/judgment/env/其他】
情境：一句話，當時在做什麼
坑：實際發生了什麼、為什麼錯
修法：下次怎麼避免（要具體到可執行）
```

寫完判斷：這條教訓是否通用到值得進正式檔？
- 是 → 按第 1 節權限改對應的 ops/ 檔，並在 lesson 條目補一行「已收編至【檔案】」。
- 不確定 → 留在 lessons.md，等精簡週期處理。

## 3. 精簡週期（防制度肥大）

觸發條件（任一）：
- `lessons.md` 超過 30 條未收編條目。
- CLAUDE.md 超過 80 行。
- 任一 ops/ 檔超過 250 行。

觸發後的動作（一個 session 內完成）：
1. 備份全部要動的檔。
2. lessons.md：重複主題合併、已收編的條目移到檔尾「已歸檔」區、明顯過時的標註棄用（不刪，標註）。
3. CLAUDE.md / ops 檔：把膨脹段落抽成新檔 + 一行路由，或合併重複規則。**規則總量只能持平或變少，不能藉精簡之名加新規則。**
4. 派 fresh-context agent 對抗審查精簡結果（規則有沒有被改變語意、路由有沒有斷），修完才算完成。
