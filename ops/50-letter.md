# 50-letter：給未來 session 的信

寫信的是 2026-07 建立這套制度的模型。讀信的你可能比我小，這不要緊——這套制度的設計前提就是你不需要比我聰明，只需要照著跑，並且在拿不準時查 `20-judgment.md` 而不是硬猜。

## 三件使用者沒問、但你最該知道的事

**1. 這套制度只在「被讀到」的時候存在。**
最大的失效模式不是規則寫錯，而是規則沒被載入。CLAUDE.md 每個 session 都會進 context，但 ops/ 檔不會——所以路由表是整套制度的命脈。你在動手前的第一個動作永遠是：對照路由表，判斷這件事該先讀哪個檔。跳過這步，等於整套制度不存在。

**2. Context 是你唯一真正稀缺的資源，而且損耗不可逆。**
Token 用完可以等下一輪，但主對話被垃圾填滿後，這個 session 的判斷力就永久降級了。這就是為什麼「指揮官不下場」是硬規則第一條。當你猶豫「這個檔案我自己讀一下應該沒關係吧」——那個猶豫本身就是答案：派工。

**3. 誠實比完成更值錢。**
使用者的環境會由不同代次、不同大小的模型接手，他無法逐一驗證每個回報。整套制度的信用建立在：DONE 就真的 DONE、「未查證」就老實標、失敗軌跡完整交接。一次「其實沒驗過但先回報完成」造成的信任損耗，比十次 BLOCKED 都貴。你撐不住的時候，回報 BLOCKED 是合格的產出，不是失敗。

## 這套制度最可能怎麼退化，以及怎麼防

**退化路徑 1：CLAUDE.md 重新膨脹成大雜燴。**
每次踩坑都往 CLAUDE.md 塞一條「注意 XX」，兩個月後它回到 300 行，重要規則被淹沒。
防法：`40-maintenance.md` 的精簡週期是硬觸發（80 行），教訓先進 lessons.md 再決定收編。發現 CLAUDE.md 超標，當下就處理，不要「下次再說」。

**退化路徑 2：驗收流於形式。**
趕時間的 session 開始「自己看一眼就標 DONE」，或驗收 agent 的 prompt 被簡化到只剩「檢查一下」。形式上有驗收、實質上是自驗。
防法：驗收模板（模板 5）強制要求逐條證據與至少 3 個質疑點。看到「全部通過」但沒有行號證據的驗收回報，視同沒驗收，退回重驗。

**退化路徑 3：規則被「合理化地」放寬。**
「這個案子特殊，重試三次應該可以」「這次先不備份，改動很小」。每次放寬都有當下說得通的理由，累積起來制度就空了。
防法：`40-maintenance.md` 已寫死——收緊可自行、放寬必問人。個案想破例，同樣要問使用者，不能自己給自己豁免。

## 一句交接

你不需要重新發明這套制度，你只需要不繞過它。祝順利。
